# Gemma 4 on Jetson Thor: A Practical Edge Benchmark for Multimodal AI

When people talk about running large multimodal models at the edge, the conversation usually drifts toward model cards, parameter counts, or generic chat demos. That is not the question that matters in the field. The real question is whether a model can do useful, inspectable work on-device under practical edge conditions: answer from local documents, read images, return structured JSON, handle multilingual content, and stay stable enough to integrate into real systems.

That is what this benchmark was built to measure.

This project evaluates Gemma 4 26B-A4B on NVIDIA Jetson AGX Thor using vLLM and the OpenAI-compatible chat endpoint. The focus is not "maximum Gemma 4 performance" across the full family. It is a deployment-focused study of the Gemma 4 Mixture-of-Experts configuration that is especially interesting for edge systems because it offers a different capability-versus-efficiency tradeoff from both the smaller Gemma 4 models and the larger dense model.

It is important to be explicit about scope. This benchmark is about Gemma 4 26B-A4B, not the entire Gemma 4 lineup. We did not run a family-wide comparison against Gemma 4 E2B, Gemma 4 E4B, or Gemma 4 31B in this repository. The goal here was narrower and more practical: determine how well the 26B-A4B MoE model behaves on Jetson Thor for grounded text, image understanding, structured outputs, multilingual extraction, and agent-style edge tasks.

The deployment under test used vLLM with the /v1/chat/completions interface, official-style Gemma 4 reasoning and tool parsers, and structured message formatting rather than flattened prompts. Prompt-token accounting also used vLLM's chat-aware tokenize path so the counts reflect the real Gemma chat template rather than an approximation. The benchmark was run with text and image enabled, audio intentionally disabled in the deployed build, a 65,536-token context window for standard workloads, a 2,096-token output cap for the main benchmark, and a pinned baseline image soft-token budget of 280, with additional image runs at 560 and 1120 for vision-focused sweeps.

The main Phase 1 run covered 33 workload families and produced 9,932 scored records. Those workloads included grounded QA, summarization, structured extraction, multi-hop reasoning, multilingual text tasks, multilingual image-text extraction, tool-use scenarios, prompt injection resistance, abstention calibration, citation tasks, and closed-set image classification benchmarks. The benchmark also captured full artifacts for later review: request payloads, visible outputs, reasoning traces when present, streaming timelines, token usage, latency, and rubric-ready records for human or LLM evaluation.

The top-line result was strong. Weighted across the full Phase 1 run, semantic task-completion accuracy reached 93.43 percent. Looking at the two operating modes separately, thinking=false delivered 95.45 percent weighted accuracy, while thinking=true delivered 91.40 percent. That alone is a useful edge result: the default non-thinking configuration was not just faster, it was also the most reliable overall configuration in this deployment.

The most impressive story in the benchmark was multimodal performance. Across the top-level image bucket, accuracy reached 96.3 percent with thinking disabled and 92.7 percent with thinking enabled. On CIFAR-10, Gemma 4 26B-A4B achieved 91.60 percent accuracy without thinking and 90.60 percent with thinking. On Caltech-256, it reached 99.60 percent without thinking and 99.93 percent with thinking. Those are not toy chat outputs. They are strong closed-set vision results produced on a Jetson Thor deployment using a general multimodal model served through a production-style API.

The multilingual image-text extraction workload was another standout. In the benchmark's OCR-style multimodal family, the model reached 97.70 percent semantic accuracy in thinking=false mode and 88.00 percent in thinking=true mode. The stronger default mode was clearly no-think, but the important result is broader than a single percentage: Gemma 4 handled multilingual image-conditioned extraction robustly across a diverse language set, which is exactly the kind of capability that matters for edge use cases such as field-service tablets, multilingual inspection workflows, warehouse tools, and offline document support systems.

The language-level OCR breakdown was especially encouraging in the default operating mode. Accuracy was near ceiling for most languages in the no-think configuration, showing that the model can serve as a practical on-device extractor when the deployment goal is reliable transcript-style output rather than long-form internal reasoning. That matters because many edge products do not want a verbose assistant. They want a model that can look at an image, extract the right thing, and return a clean machine-consumable answer.

Text and reasoning tasks were also strong in several categories that matter operationally. Structured extraction reached 90.00 percent without thinking and 96.67 percent with thinking. Multi-hop reasoning reached 100.00 percent in both modes. Temporal sequence reasoning also reached 100.00 percent in both modes. Prompt injection resistance scored 100.00 percent in both modes, and abstention calibration also scored 100.00 percent in both modes. These are exactly the kinds of results that make a model useful beyond casual chat. They point toward a system that can work with grounded inputs, maintain schema discipline, and behave predictably under instructions that matter in production.

One of the most useful findings from the benchmark was not that thinking mode is bad, but that it should be used selectively. Thinking=true did help in some places, especially structured extraction, and it can be valuable when the workload benefits from richer intermediate reasoning. But for the benchmark as a whole, thinking=false was the better default. It produced the best balance of accuracy, latency, and completion reliability. That is a practical deployment lesson. On the edge, the best model configuration is not the one that appears most sophisticated. It is the one that delivers the best end-to-end behavior for the task mix you actually care about.

The latency data reinforced that conclusion. Median latency for CIFAR-10 was 1.43 seconds with thinking disabled versus 8.30 seconds with thinking enabled. Caltech-256 was 1.45 seconds versus 7.26 seconds. Multilingual image-text extraction was 1.95 seconds versus 10.25 seconds. Grounded QA was 1.63 seconds versus 13.13 seconds. Multi-hop reasoning was 3.64 seconds versus 12.36 seconds. In other words, Gemma 4 on Jetson Thor looked strongest when configured as a concise, grounded, machine-facing multimodal assistant rather than as a high-deliberation reasoning engine for every request.

That is not a weakness. It is exactly the kind of deployment insight edge teams need.

If you are building for offline or low-connectivity environments, the capability mix shown here is promising. The benchmark suggests a strong fit for use cases such as document-grounded support assistants, field-service copilots, multilingual OCR and extraction tools, visual classification and inspection interfaces, kiosk or tablet-based operations support, and structured-output workflows where the model needs to slot into software rather than simply chat with a person.

The practical appeal of Gemma 4 26B-A4B in this context is that it combines several qualities that are often hard to get together on-device. It is multimodal. It can return structured outputs. It can follow tool-oriented chat patterns through a standard API. It can handle multilingual image-text extraction. And, in the right operating mode, it can do all of that with a latency profile that makes edge interaction plausible rather than theoretical.

Another important part of this work is methodology. The benchmark was designed to generate evidence that can be audited later, not just a chart for a slide. Each scenario is paired with a rubric. Structured tasks are checked against exact or constrained fields. Closed-set image tasks are scored against labels. OCR-style tasks require transcript fidelity plus correct language handling. Prose tasks are checked against required facts and prohibited behaviors. The result is a benchmark that is useful not only for marketing summaries, but also for engineering review and future model comparison.

That matters because edge AI deployment is not just about model quality. It is about trust. If a model is going to be used in field operations, manufacturing support, local document search, inspection workflows, or multimodal assistants running without internet connectivity, teams need to know what it does well, what settings make it stable, and what outputs can be audited after the fact.

The headline answer from this Jetson Thor study is that Gemma 4 26B-A4B looks genuinely capable for that class of work.

The strongest public takeaway is not that the model solved every hard problem in the repository. It did not. The strongest takeaway is that it delivered very strong multimodal and structured-task performance on-device, with especially compelling results on image classification, multilingual image-text extraction, structured extraction, multi-hop reasoning, temporal reasoning, prompt injection resistance, and abstention behavior. For the kinds of edge systems that need reliable text-plus-image understanding and clean machine-consumable outputs, that is a meaningful result.

If I were summarizing the deployment recommendation in one sentence, it would be this: Gemma 4 26B-A4B on Jetson Thor is most compelling today as a grounded multimodal edge assistant running through vLLM with thinking disabled by default and selectively enabled only for tasks that clearly benefit from deeper reasoning.

That is a useful place to be. It means the model is not just interesting in theory. It is useful in the shape that real edge applications actually need.
